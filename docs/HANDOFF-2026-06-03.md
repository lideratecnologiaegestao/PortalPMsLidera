Scaffold com expo-router: lista de próximos, novo chamado (câmera+GPS+anônimo), acompanhar por protocolo, cliente de API. **Não buildado** (precisa `npm install` + `expo start`).

## 2. Como rodar e testar (ambiente local)

> Docker só existe no WSL; o `postgres-main` ocupa a 5432. Por isso o dev usa containers dedicados em portas alternativas.

### Stack online (o que está rodando agora)
- **Postgres+PostGIS**: container `portal-postgres-dev` em `127.0.0.1:5433` (todas as migrations aplicadas).
- **Redis**: container `portal-redis-online` em `127.0.0.1:6380`.
- **API**: `node dist/main.js` em `:3001` (ver env abaixo).
- **Web**: `next start -p 3000` em `:3000`.
- **Tenant de demonstração**: `infra/seed-demo.sql` (domínio `localhost:3000`, dados em todos os módulos, IA habilitada, diário publicado com hash/assinatura coerentes).

Acesse **http://localhost:3000** (use `localhost`, não `127.0.0.1` — o tenant é resolvido pelo Host).

### Subir do zero
```powershell
# 1. Banco + Redis (WSL docker)
wsl docker run -d --name portal-postgres-dev -p 5433:5432 -e POSTGRES_DB=portal -e POSTGRES_USER=portal -e POSTGRES_PASSWORD=portal postgis/postgis:16-3.4
wsl docker run -d --name portal-redis-online -p 6380:6379 redis:7-alpine
# migrations (em ordem) + seed
wsl docker cp ./db portal-postgres-dev:/migrations
# psql -f /migrations/001..013 (glob ordena)
wsl docker cp ./infra/seed-demo.sql portal-postgres-dev:/tmp/seed.sql
wsl docker exec portal-postgres-dev psql -U portal -d portal -f /tmp/seed.sql

# 2. API
cd api; npm install; npx prisma generate; npm run build
$env:DATABASE_URL="postgresql://portal:portal@127.0.0.1:5433/portal"
$env:REDIS_HOST="127.0.0.1"; $env:REDIS_PORT="6380"; $env:REDIS_DB="1"
$env:AUTH_JWT_SECRET="<32+ chars>"; $env:CPF_PEPPER="<64 hex>"; $env:DIARIO_SIGNING_KEY="dev-diario-key-online"
node dist/main.js

# 3. Web (NEXT_PUBLIC_API_BASE assa no build)
cd web; npm install
$env:NEXT_PUBLIC_API_BASE="http://localhost:3001"; npm run build
$env:API_URL="http://127.0.0.1:3001"; node node_modules/next/dist/bin/next start -p 3000
```

> O boot da API pode levar ~20s por retries de conexão do Redis no WSL (flakiness do ambiente, não do código).

### Testes
```powershell
cd api
$env:DATABASE_URL="postgresql://portal:portal@127.0.0.1:5433/portal"
$env:AUTH_JWT_SECRET="<32+>"; $env:CPF_PEPPER="<64 hex>"
npm test            # unit (50): FSM, SLA, PKCE, JWT, confiabilidade, CSV, máscara, EXIF, IA prompts
npm run test:e2e    # e2e (20, runInBand): isolamento RLS por módulo (conecta como portal_app)
```
**Total: 70 testes.** Os e2e conectam como `portal_app` (NOSUPERUSER) porque superusuário ignora RLS.

## 5. Bloqueantes restantes (antes do go-live)
Revisão dupla (segurança + qualidade) resolveu todos os bloqueantes **técnicos**. Sobram itens **não fecháveis por código**:
- **DPA com a Anthropic** (transferência internacional) — contratual.
- **Certificado ICP-Brasil** real para o Diário — o `SignatureService` já barra publicação em produção sem `ICP_CERT_PATH`.
- **Antivírus (ClamAV)** nos uploads públicos; **MFA forte** para servidores (hoje o gate de confiabilidade gov.br é proxy).

Detalhe completo do que foi corrigido vs. pendente: memória `prod-readiness-backlog`.
